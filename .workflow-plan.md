# Implementation plan — Issue #55: PR-CI admin production-image boot smoke

Branch: `feat/55-admin-image-smoke-ci`
Linked roadmap item: <https://github.com/caelo-cms/caelo-cms/issues/55>

## 1. Goal

Close the cheaper half of the gap #53 exposed: a PR whose change breaks the
admin's production Docker image — specifically, the boot path (a
`Cannot find module "oxc-parser"`-class regression) — currently has only one
gate, the `release-images` workflow added by #54. `release-images` exercises
the full publish pipeline (build → push → cosign sign → mirror) and is the
right gate for the "image actually builds and uploads" question, but it is
heavy (matrix admin + gateway, ~6 min warm) and only verifies BUILD success,
not BOOT. The bug class #53 fixed is a runtime crash at module load — the
Docker build itself succeeds; the container then dies on `docker run` because
the externalized `oxc-parser` import can't be resolved from the trimmed image
tree.

This PR ships a complementary, lighter PR-CI job — `admin-prod-image` — that
builds the admin image alone (single-arch, no push, no sign, no mirror, GHA
layer cache shared with release-images) and `docker run`s it to verify the
SvelteKit + adapter-bun server actually boots without throwing. Job lives in
the main `ci.yml` workflow (not `release-images.yml`) because it is a
correctness-of-the-prod-image-at-runtime check, not a publish step. It runs
in parallel with the existing `release-images` matrix; the two are
complementary (release-images catches build-time regressions; admin-prod-image
catches boot-time regressions) and both stay green on the same PR.

## 2. Scope (in)

- **`.github/workflows/ci.yml`** — add a new job `admin-prod-image` (display
  name `Admin production image — boot smoke`) under the `jobs:` block. The job
  runs on every PR + push to main (the workflow's existing `on:` triggers
  cover both). Shape:
  - `needs: lockfile` — same gate the `check` and `e2e` jobs use; no point
    building the image if the lockfile is dirty.
  - `runs-on: ubuntu-latest`, `timeout-minutes: 10` — generous ceiling, but
    the steady-state target is <90 s (AC #3).
  - Steps (top-to-bottom):
    1. `actions/checkout@v4` — fetch source.
    2. `docker/setup-buildx-action@v3` — for `cache-from`/`cache-to` GHA cache.
    3. `docker/build-push-action@v6` with:
       - `context: .`
       - `file: apps/admin/Dockerfile`
       - `platforms: linux/amd64` — matches release-images; ARM is deferred
         per the workflow's existing comment.
       - `load: true` — required for `docker run` in the next step (otherwise
         the image lives only in the buildx cache and `docker run` can't see
         it).
       - `push: false` — never push from this job; the publish path is
         release-images.
       - `tags: caelo-admin:smoke` — local tag for the subsequent run step.
       - `cache-from:` reads BOTH `type=gha,scope=admin` (the release-images
         admin matrix job's cache) AND `type=gha,scope=admin-smoke` (our own
         prior runs).
       - `cache-to: type=gha,scope=admin-smoke,mode=max` — write to our own
         scope so we never thrash release-images' cache. Buildx supports
         multiple `cache-from` entries; we use that to share the release-
         images cache one-way (read-only) without contention.
    4. `Boot smoke` — a single `run:` block (with `set -euo pipefail` at
       the top so a `docker run` / `curl` / `docker inspect` failure
       surfaces as a job failure rather than being silently swallowed)
       that:
       - `docker run -d --name smoke -p 3000:3000 -e PORT=3000 \
         -e ORIGIN=http://localhost:3000 \
         -e ADMIN_DATABASE_URL=postgres://nobody:nothing@127.0.0.1:9/nope \
         -e PUBLIC_DATABASE_URL=postgres://nobody:nothing@127.0.0.1:9/nope \
         caelo-admin:smoke`
         — fake DB URLs are intentional. `apps/admin/src/lib/server/query.ts`
         constructs the `DatabaseAdapter` lazily on first request, not at
         module-load time; per-request `bootstrap*` calls in
         `apps/admin/src/hooks.server.ts` are all wrapped in `.catch(...)`
         so a DB connection refusal logs but does not crash the process.
         The `MODULE_NOT_FOUND` class of bug fires at module load, BEFORE
         the server starts listening — so a successful port-bind already
         confirms the boot path is healthy. AC #4 explicitly accepts
         boot-only verification at minimum.
       - Wait up to 30 s for the server to accept any HTTP response on
         `/` (success = a `^[0-9]{3}$` HTTP status from curl; even 500 is
         a valid signal that the JS bundle loaded and the listener
         attached). Implemented as a `while [ $SECONDS -lt $deadline ]`
         polling loop with `curl --max-time 2`.
       - On timeout: `docker logs smoke` + `exit 1` with a `::error::`
         annotation pointing at the boot failure.
       - On success: assert `docker inspect -f '{{.State.Running}}' smoke`
         is `true` (catches the rare "bound the port then crashed" case).
       - Capture and print `docker logs smoke | head -50` on success too,
         so the run log shows the admin's structured-log lines (useful
         for debugging future flakes).
       - `docker rm -f smoke` at the end so the runner is clean.
  - The 30 s wait deadline + 2 s per-curl timeout gives ~15 polling attempts.
    Once the server is up, the first attempt resolves <100 ms. Cold-start
    measurements from local dev runs against the same image: ~5 s to bind
    PORT, ~8-12 s for `bootstrapPlugins`'s manifest scan (which we don't
    block on). 30 s ceiling is generous.

- **`.github/rulesets/main.json`** — append one entry to
  `rules[].parameters.required_status_checks`:
  - `{ "context": "Admin production image — boot smoke" }`
  This is the rendered `name:` field of the new job; GitHub reports the
  rendered name as the check context. The existing two release-images
  contexts (`build + push (admin)`, `build + push (gateway)`) stay; this PR
  adds the boot-smoke as a third Docker-related required check.

- **`CONTRIBUTING.md`** — extend the "Required CI checks must pass before
  merge" bullet inside the `## Pull request mechanics` section to add the
  new check name with a one-line rationale ("catches `Cannot find module`-
  class regressions that succeed at Docker build time but crash on
  `docker run` — the boot half of the `#53` bug class; complements the
  `build + push (admin)` gate which catches the build half"). Anchor by
  section name; no line-number coupling.

- **`scripts/admin-image-boot-smoke.test.ts`** — NEW. Pure `bun:test` over
  `.github/workflows/ci.yml` and `.github/rulesets/main.json`. Mirrors the
  shape of `scripts/release-images-pr-gate.test.ts` (#54's precedent) and
  `scripts/dockerfile-bunfig-copy.test.ts`. Picked up automatically by
  `bun test` in the `check` job — no CI YAML edit needed to wire it in.
  Assertions (numbered to track this plan's Tier 1 test strategy below):
  - C1: `ci.yml` parses cleanly as YAML (defense in depth so a structurally
    broken workflow fails here, not at GitHub Actions upload time).
  - C2: a job keyed `admin-prod-image` exists under `jobs:` and its `name:`
    field is exactly `Admin production image — boot smoke` (the name is
    the GitHub check context, so a rename would silently drop the
    required-check binding).
  - C3: the job's `needs:` includes `lockfile` (mirrors `check`/`e2e`).
  - C4: the job's steps include a `docker/build-push-action@v6` step (or a
    later major when bumped) whose `with.file` is `apps/admin/Dockerfile`
    and whose `with.push` is `false` and whose `with.load` is `true`.
  - C5: the job's steps include a step whose `run:` body contains
    `docker run` AND `--name smoke` AND a port-wait loop (substring match
    on a `while ` AND `curl ` — both must appear in the same `run:`).
  - C6: the job's steps include a `docker logs smoke` invocation (the
    failure-path log dump; a regression that removes it makes future
    flakes opaque).
  - R1: the ruleset's required_status_checks list contains
    `Admin production image — boot smoke`.
  - R2: the ruleset still contains the existing four required contexts
    (`Lockfile freshness`, `Lint, Typecheck, Migrate, Test, License`,
    `build + push (admin)`, `build + push (gateway)`). Defense in depth
    against a hand-edit that adds ours but drops one of the existing
    checks.

  Test uses `js-yaml` (already in devDependencies — release-images-pr-gate
  test imports it) for the YAML parse and `JSON.parse` for the ruleset.
  Slice helpers (`extractJobBody`, etc.) mirror release-images-pr-gate.

## 3. Out of scope (later)

- **Postgres `services:` + real `GET /` assertion.** AC #4 says full
  `GET /` is a "bonus"; the minimum is boot-only verification. Adding a
  Postgres service + bootstrap.sh + migrations adds ~30-40 s to the job
  (pushes us toward the 90 s ceiling) for a marginal value-add over what
  the existing `e2e` Playwright job already covers (the e2e job runs
  against the production bundle with a real DB and asserts the login
  redirect). The boot-only smoke is the load-bearing piece this PR is
  shipping. A follow-up issue can add a `GET /` assertion if the
  observed value justifies the cost.
- **Gateway image boot smoke.** The gateway image is a thinner shell
  (`apps/api-gateway/Dockerfile`) with no SvelteKit / vite bundle, no
  oxc-parser dispatcher, and no per-request bootstrap chain. The bug
  class this PR addresses is admin-specific. A symmetric `gateway-prod-
  image` job is fine in principle but adds CI minutes for a surface
  that has not regressed; defer until justified by evidence.
- **Apply the ruleset to GitHub.** Editing `.github/rulesets/main.json`
  changes the source-of-truth file; the live ruleset on github.com is
  only updated when a maintainer runs `bun run rulesets:apply` (per the
  file header comment and the precedent from #54). This PR commits the
  JSON change; the maintainer running `apply` is a separate, post-merge
  step. The PR description must flag this — without the apply run, the
  new context is present in the file but not enforced by GitHub.
- **Cache-eviction tuning.** GHA's cache has a 10 GB limit per repo;
  `scope=admin-smoke` adds another scope alongside `scope=admin`. We
  rely on GitHub's LRU policy. A measurable cache-thrash regression
  would justify consolidating scopes; defer until observed.
- **AI security review on the workflow change itself.** The existing
  `security-review.yml` workflow auto-reviews every PR; nothing to wire
  up. The plan + test file fall under its review window.
- **Multi-arch (linux/arm64) build.** Out of scope per the workflow's
  existing comment (multi-arch waits on Bun's QEMU SIGILL on arm64,
  bun#15044).

## 4. Architecture impact

- **Layer:** CI / build tooling only. No permission layer, no Query API,
  no DB schema, no SDK surface, no plugin code, no admin UI, no migration.
- **CLAUDE.md invariants honoured:**
  - §9 "Commits, PRs, and reviews" — "Required CI checks must pass before
    merge." This PR adds one more check to that list, directly serving
    the invariant.
  - §11.B "Deployment architecture per provider" — Tier 2 admin app
    correctness is the same shape across providers; the boot smoke
    verifies the image that every provider's Cloud Run / Container Apps /
    Fargate / Caddy installs.
  - §11.C "Provisioning UX — one command contract" — "Pre-built container
    images on a public registry. Users do not build images." The PR
    protects the only image users consume by catching boot-time
    breakages pre-merge instead of post-merge / post-deploy.
  - §3 "Dependency & version policy" — no new dependency added (the test
    file reuses `js-yaml`, already present for `release-images-pr-gate.test.ts`).
  - §6 "Testing" — three-tier strategy honoured (see § "Test strategy"
    below); Tier 2 + Tier 3 are deliberately empty with rationale, same
    pattern as #54.
  - §7 "Security posture" — the boot smoke uses fake DB URLs that
    cannot reach any real database; the container is killed at the end
    of the job. No secrets, no network egress beyond the runner.
  - **Fork-PR safe by design.** The job references no `secrets.*`
    values, uses no credential-providing action (`docker/login-action`,
    `sigstore/cosign-installer`, `google-github-actions/auth`), and
    builds with `push: false`. Fork PRs run it the same as base-repo
    PRs without any `if:` gates — the gating dance #54 had to do for
    `release-images.yml` doesn't apply here.
- **CMS_REQUIREMENTS.md §6 (Deployment & Hosting) + §17.4 (AI Restrictions)** —
  the release-images workflow + this PR's CI job are the Ops mechanism for
  shipping signed images and verifying their runtime correctness. §17.4
  locks AI runtime actors out of deploy/CI logic; this PR is human-
  authored (modifying `.github/workflows/*.yml` is by definition outside
  the runtime AI's reach) and adds a runtime-correctness gate without
  changing what publishes or how.

## 6. Risks & open questions

- **Risk: the check context name reported by GitHub differs from the
  job's `name:` field.** GitHub reports the rendered job `name:` as the
  required-check context; for non-matrix jobs this is straightforward,
  but quirks (e.g. the workflow name being prefixed) have bitten before.
  **Mitigation:** verify on the first PR run that the context appears
  exactly as `Admin production image — boot smoke` in the GitHub PR
  checks UI. If GitHub prefixes the workflow name (e.g. `CI / Admin
  production image — boot smoke`), update `main.json` to match. The
  `bun run rulesets:check` script catches drift between the file and
  the live state, so a mismatch surfaces quickly.
- **Risk: 30 s boot deadline is too tight on a cold runner.** Empirical
  cold-start measurements indicate ~5-12 s to bind PORT; 30 s gives 2.5x
  headroom. If observed flake rate exceeds 1 in 50 runs, bump to 60 s.
  Documenting this risk so a future maintainer reading a flake post-mortem
  finds the tuning knob.
- **Risk: cache-from `scope=admin` is empty on the very first PR
  (release-images hasn't yet completed for this branch).** First run on
  a new branch is cold for both jobs in parallel; subsequent pushes
  warm the per-scope caches independently. **Mitigation:** none needed —
  first-run cost is one-time and bounded by the 10-minute job timeout.
- **Risk: the boot-only verification gives a false sense of confidence
  for DB-touching regressions.** A regression that breaks the
  `auth.resolve_session` op path wouldn't show up here. **Mitigation:**
  the existing `e2e` Playwright job covers that surface against a real
  DB and a real `GET /login`. The two jobs are complementary — this PR's
  job is specifically the boot-time / runtime-resolution check, NOT a
  full-stack check.
- **Risk: future contributor adds a credential-touching step to the
  new job and exposes a secret to fork PRs.** **Mitigation:** the job
  body contains no `secrets.*` references and uses no credential-
  providing actions. The release-images-pr-gate test's W6 assertion
  is defense in depth for `release-images.yml`; if needed, an
  equivalent assertion can be added against `ci.yml` in a follow-up.
  This PR ships without it because the job is deliberately credential-
  free by design.
- **Open question: should the job use `--network host` and a real
  Postgres service?** **Decision:** no, per the scope/out-of-scope
  split above. AC #4 accepts boot-only as the minimum, AC #3 favors
  leaner, and the existing `e2e` job already covers the real-DB
  surface. Reopen this question if the boot-only smoke fails to catch
  a runtime regression that a full-stack smoke would have caught.
- **Decision (resolved in plan §2): the job lives in `ci.yml`, not
  `release-images.yml`.** Rationale: `release-images.yml`'s job names
  are matrix-rendered (`build + push (admin)`) and conflate "build the
  image" with "publish the image to GHCR + sign + mirror"; injecting a
  boot step into that matrix entangles two concerns. A separate
  top-level job in `ci.yml` keeps the boot smoke discoverable, gives
  it its own clean check context, and lets it run with its own
  concurrency / cache scopes.

- [ ] CI's `admin-prod-image` job runs on every PR opened against `main`.

- [ ] At minimum, asserts the server process boots without throwing
      (full `GET /` is a bonus). (Satisfied by the boot-wait + alive
      check; `GET /` is exercised by the curl polling loop and its

- [ ] Job is in the required-checks list on the main branch ruleset.
      (Satisfied by appending the context to
      `.github/rulesets/main.json` — final enforcement requires a

### Manual production-bundle smoke (step 09)

Encoded so step 09 (verify-locally) has an exact recipe; not part of CI,
but part of the workflow's per-step verification per CLAUDE.md §6:

```sh
# 1. Lint + typecheck — catches structural YAML mistakes (bad indentation,
#    unclosed mapping) that the bun test can't and verifies the new test
#    file typechecks against the workspace.
bun run lint
bun run typecheck

# 2. Run the new test file directly to confirm it passes before pushing.
bun test scripts/admin-image-boot-smoke.test.ts

# 3. Build the admin image locally with the same args CI uses, and run
#    the boot smoke. This is the SAME recipe the new job runs on CI —
#    useful for tuning the 30s wait deadline if local boot is slow.
DOCKER_BUILDKIT=1 docker build -f apps/admin/Dockerfile -t caelo-admin:smoke .
docker run --rm -d --name smoke -p 3000:3000 \
  -e PORT=3000 -e ORIGIN=http://localhost:3000 \
  -e ADMIN_DATABASE_URL=postgres://nobody:nothing@127.0.0.1:9/nope \
  -e PUBLIC_DATABASE_URL=postgres://nobody:nothing@127.0.0.1:9/nope \
  caelo-admin:smoke
# Wait for /, observe HTTP code (any 3-digit response means boot OK)
for i in $(seq 1 30); do
  code=$(curl -sS -o /dev/null -w '%{http_code}' --max-time 2 http://localhost:3000/ || true)
  [ -n "$code" ] && [ "$code" != "000" ] && echo "boot OK: HTTP $code" && break
  sleep 1
done
docker logs smoke | head -20
docker rm -f smoke

# 4. Dry-run the ruleset apply to confirm the JSON's shape is valid
#    against GitHub's expected schema.
bun run rulesets:apply --dry-run
# Expect: a single `[dry-run] would update main-protection from main.json` line.
```
