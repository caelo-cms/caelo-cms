# Implementation plan — Issue #54: run `release-images` on PRs and make it a required check

Branch: `feat/54-release-images-pr-gate`
Linked roadmap item: <https://github.com/caelo-cms/caelo-cms/issues/54>

## 1. Goal

Close the regression class that #53 exposed: a PR whose change breaks the
production admin Docker image still passes CI today because the only workflow
that exercises the lean production image (`release-images.yml`) runs only on
push-to-`main` — i.e. *after* merge. Shipping this PR adds `pull_request` to
that workflow's triggers so the build runs on every PR against `main`, gates
all credential-dependent steps (GHCR login, image push, cosign sign/verify, GCP
Artifact Registry mirror) behind an `if:` so fork PRs can't fail on missing
secrets, and adds the two matrix jobs (`build + push (admin)` and
`build + push (gateway)`) to the `main-protection` ruleset's required-status
list so a PR with a broken Docker build cannot be merged. CONTRIBUTING.md is
updated so the new gate is discoverable to contributors.

## 2. Scope (in)

- **`.github/workflows/release-images.yml`** — add
  `pull_request: { branches: [main] }` AND `merge_group:` to the `on:` block
  alongside the existing `push`, `workflow_call`, `workflow_dispatch`.
  `merge_group:` is zero-cost today and prevents a follow-up PR when #34
  enables GitHub's native merge queue (the queue re-runs required checks on
  the merge group; without the trigger the gate would silently skip the
  queue). Update the top-of-file comment to record that the workflow now
  runs on PRs (build-only) as well as the existing post-merge / release
  paths.
- **`.github/workflows/release-images.yml`** — add a `concurrency:` block at
  the workflow level:
  `group: release-images-${{ github.workflow }}-${{ github.event.pull_request.number || github.ref }}`
  with `cancel-in-progress: ${{ github.event_name == 'pull_request' }}`. A
  rapid `git push` to a PR branch supersedes the prior build's relevance;
  cancelling the in-flight job saves ~6 min of CI minutes per push. Push-to-
  main builds keep `cancel-in-progress: false` so a release-bound build is
  never interrupted by a follow-up commit.
- **`.github/workflows/release-images.yml`** — gate the side-effect steps with
  `if: github.event_name != 'pull_request'`:
  - "Log in to GHCR" (no push → no login needed).
  - "Install cosign" / "Sign image with cosign (keyless / OIDC)" / "Verify the
    signed image (sanity check)" — keyless signing needs the `id-token` OIDC
    token that fork PRs don't get, and there's no image to sign anyway.
  - "Authenticate to GCP via Workload Identity Federation" / "Configure docker
    for GCP AR push" / "Mirror image (+ signatures) to GCP public AR" — the
    existing `vars.GCP_PUBLIC_REGISTRY_WIF_PROVIDER != ''` guard becomes an
    AND with the new event guard.
- **`.github/workflows/release-images.yml`** — flip the `Build + push` step
  from unconditional `push: true` to
  `push: ${{ github.event_name != 'pull_request' }}` so the build step still
  runs on PRs (this is the load-bearing assertion) but doesn't attempt a
  registry write without credentials. Keep `cache-from`/`cache-to` as-is — GHA
  caches are branch-scoped, so a PR cache can't pollute `main`'s.
- **`.github/rulesets/main.json`** — append two entries to
  `rules[].parameters.required_status_checks`:
  - `{ "context": "build + push (admin)" }`
  - `{ "context": "build + push (gateway)" }`
  These are the matrix-expanded job names from `release-images.yml`. The
  ruleset's `strict_required_status_checks_policy: true` already forces the PR
  branch to be up-to-date with `main` before merge, so the gate is meaningful.
- **`CONTRIBUTING.md`** — extend the "Required CI checks must pass before
  merge" bullet inside the `## Pull request mechanics` section to add the two
  new check names, with a one-line rationale ("catches regressions that only
  surface in the lean production Docker image — the same class of bug as
  #53"). Adjust the surrounding sentence as needed. Anchor by section name
  rather than line number so future edits to the doc don't invalidate this
  pointer.
- **`scripts/oxc-parser-bundle.test.ts` precedent → new
  `scripts/release-images-pr-gate.test.ts`** — pure string assertions over the
  two source files (the workflow YAML + the ruleset JSON). Confirms the gates
  haven't been silently removed. Picked up automatically by `bun test` in the
  `check` job (no CI YAML edit needed). Five assertions:
  - W1: the workflow's `on:` block contains `pull_request:` (regression that
    drops it makes the gate non-functional and resurrects #53's bug class).
  - W2: every credential-touching step (`Log in to GHCR`, `Install cosign`,
    `Sign image with cosign`, `Verify the signed image`, `Authenticate to GCP`,
    `Configure docker for GCP AR push`, `Mirror image`) carries
    `github.event_name != 'pull_request'` in an `if:` (regression that removes
    one breaks fork PRs that don't have secrets).
  - W3: the `Build + push` step uses
    `push: ${{ github.event_name != 'pull_request' }}` (regression that
    flips it back to `push: true` makes fork PRs fail at the push step).
  - R1: parse `.github/rulesets/main.json` with `JSON.parse(...)`, walk to
    `rules[type=required_status_checks].parameters.required_status_checks`,
    and assert the resulting context-name set is a superset of
    `{ "build + push (admin)", "build + push (gateway)" }` (regression that
    removes them means a broken Docker build no longer blocks merge).
  - R2: same parse, assert the set still contains the existing two contexts
    (`Lockfile freshness`, `Lint, Typecheck, Migrate, Test, License`) —
    defensive, so a future hand-edit doesn't accidentally drop one of the
    existing required checks while adding ours.

  W1–W3 use `readFileSync` + substring/regex matching on the YAML (no YAML
  parser dep, consistent with `scripts/oxc-parser-bundle.test.ts`); R1–R2
  use `JSON.parse` because the ruleset is JSON and the parsed shape is
  small + well-defined.

## 3. Out of scope (later)

- **Apply the ruleset to GitHub.** Editing `.github/rulesets/main.json`
  changes the source-of-truth; the live ruleset on github.com is only updated
  when a maintainer runs `bun run rulesets:apply` (per the file header
  comment). This PR commits the JSON change; the maintainer running `apply`
  is a separate, post-merge step. The PR description must flag this — without
  the apply run, the new contexts are present in the file but not enforced by
  GitHub.
- **Production image smoke job on PR CI (#55).** That's the sibling issue:
  build the admin Dockerfile AND boot the resulting container AND GET `/`.
  This PR ships build-only; #55 adds the boot/smoke layer.
- **Multi-arch builds, AWS ECR / Azure ACR mirrors.** Those are out of scope
  per the workflow's existing comments (multi-arch waits on bun#15044; AWS/
  Azure mirrors land alongside P15 cloud adapters).
- **AI security review on the workflow change itself.** The existing
  `security-review.yml` workflow auto-reviews every PR; nothing to wire up.
- **Tagging the new contexts as `do_not_enforce_on_create: true`** to soften
  the rollout if the workflow flakes early. The existing two required checks
  use `false`; aligning is simpler than introducing a new policy axis.

## 6. Risks & open questions

- **Risk: the matrix job names reported to GitHub differ from
  `build + push (admin)` / `build + push (gateway)`.** Matrix-with-explicit-
  `name:` typically reports the rendered `name:` field as the check context,
  but GitHub has subtle quirks (e.g. extra service-tag suffixing). **Mitigation:**
  verify on the first PR run that the contexts appear as expected in the
  GitHub PR checks UI. If they appear differently (e.g. `build + push /
  (admin)` or `release-images / build + push (admin)`), update the JSON to
  match the exact strings GitHub uses. The `bun run rulesets:check` script
  catches drift between the file and the live state, so a mismatch surfaces
  quickly. The required-check entries can also be added as best-effort first
  and refined in a follow-up commit on this same branch.
- **Risk: build time on PRs.** Each Dockerfile build runs ~5-10 min serial; the
  matrix runs admin + gateway in parallel. PR latency climbs by that much on
  every push. **Mitigation:** GHA layer cache (already configured) trims
  repeated rebuilds to <2 min once warmed; the trade-off (catching a #53-class
  regression once per release cycle vs. ~6 min added per PR) is heavily in
  favour of catching the regression. No tuning needed pre-launch.
- **Risk: ruleset apply is a manual step.** Committing the JSON does not
  enforce the check; a maintainer must run `bun run rulesets:apply` after
  merge. **Mitigation:** call this out in the PR description prominently;
  consider tracking automated apply via a CI job in a follow-up issue (out of
  scope here).
- **Decision (resolved in plan §2): `merge_group:` is included.** GitHub's
  merge queue (if enabled later, see #34) re-runs required checks on the
  merge group; without the trigger the new gate would silently skip the
  queue. Zero cost to include now.
- **Decision (resolved in plan §2): `concurrency:` block is included.** Saves
  ~6 min of CI minutes per repeated PR push. Push-to-main builds set
  `cancel-in-progress: false` to protect release-bound runs.
- **Open question: should we filter the trigger by paths?** A docs-only PR
  doesn't change the Docker image; running a 6-min build on every README edit
  is wasteful. **Decision:** out of scope for this PR — the issue asks to
  "run on every PR opened against main", and path-filtering an *acceptance-
  criteria-required* gate risks the gate silently skipping a PR that actually
  did break the build. Path filters can land in a follow-up after we
  measure the false-positive frequency in practice.

## 7. Acceptance criteria (from #54, lightly normalised)

- [ ] `release-images` workflow runs on every PR opened against `main`.

- [ ] PRs that would break the admin Docker build cannot merge — the status

      contributors. (Satisfied by extending the "Required CI checks" paragraph
      with the two new check names.)

## Test strategy

CLAUDE.md §6 requires three tiers per phase. This PR's surface is CI
configuration + a ruleset JSON + a docs paragraph + one regression test —
no runtime code path, no Query API op, no admin UI behaviour. The strategy
below names every assertion file + assertion case so a reviewer can predict
which regressions are caught (and which deliberately are not).

### Tier 1 — Unit (`bun test`, in the `check` CI job)

**New file:** `scripts/release-images-pr-gate.test.ts`.

Shape mirrors `scripts/oxc-parser-bundle.test.ts` (#53's precedent) — pure
`bun:test`, no Postgres, no Playwright, no I/O beyond `readFileSync` + a
single `JSON.parse`. Runs in <50ms.

Reads two source files at suite start:

- `workflow = readFileSync('.github/workflows/release-images.yml', 'utf8')`
- `ruleset = JSON.parse(readFileSync('.github/rulesets/main.json', 'utf8'))`

Assertions, each with the exact regression it catches:

| # | Assertion | File | Catches |
|---|---|---|---|
| W1 | The `on:` block contains both `pull_request:` and `merge_group:` triggers (regex scoped to the substring window between `^on:$` and the next top-level key). | `release-images.yml` | A regression that removes the `pull_request` trigger restores the #53 bug class (broken Dockerfile rides into main); removing `merge_group` makes the gate silently skip under GitHub's merge queue once #34 enables it. |
| W2 | Every step whose `name:` matches one of (`Log in to GHCR`, `Install cosign`, `Sign image with cosign (keyless / OIDC)`, `Verify the signed image (sanity check)`, `Authenticate to GCP via Workload Identity Federation`, `Configure docker for GCP AR push`, `Mirror image (+ signatures) to GCP public AR`) carries `github.event_name != 'pull_request'` in an `if:` clause. Implemented as: for each name, find the substring window from `- name: <name>` to the next `- name:` (or end-of-file) and assert the gate token is present within that window. | `release-images.yml` | A regression that drops the gate from any one step breaks fork PRs: that step tries to use secrets the fork-PR runner doesn't carry and the whole job goes red even though the build itself succeeded. AC #3 fails. |
| W3 | The `Build + push` step's `with.push:` parameter is the expression `${{ github.event_name != 'pull_request' }}` (substring match scoped to the `- name: Build + push` window). | `release-images.yml` | A regression that flips this back to `push: true` causes fork PRs to fail at the push step. A regression that flips it to `push: false` removes the post-merge publish — main builds would stop pushing to GHCR. The exact expression catches both. |
| W4 | The workflow has a top-level `concurrency:` block whose `group:` references `github.event.pull_request.number \|\| github.ref` AND whose `cancel-in-progress:` references `github.event_name == 'pull_request'`. | `release-images.yml` | A regression that drops the concurrency block returns to the wasteful old behaviour of running every push-to-PR through the full build serially. A regression that changes `cancel-in-progress` to a bare `true` would let a follow-up push to `main` cancel a release-bound build mid-flight. |
| W5 | The top-of-file comment block mentions PR-trigger behaviour (`grep` for the literal phrase `runs on PRs` OR `pull_request` inside the leading `#`-prefixed comment lines). | `release-images.yml` | A regression that drops the explanatory comment from the top of the file. Without the comment, a future maintainer reading the file may think the PR trigger is accidental and remove it. Defense in depth for AC #1. |
| R1 | `ruleset.rules.find(r => r.type === 'required_status_checks').parameters.required_status_checks` is an array whose element `context` values form a set that includes both `build + push (admin)` AND `build + push (gateway)`. | `main.json` | A regression that removes either context means a broken Docker build no longer blocks merge — AC #2 fails. |
| R2 | The same array's contexts also still include `Lockfile freshness` AND `Lint, Typecheck, Migrate, Test, License`. | `main.json` | A regression that drops one of the existing required checks while adding ours would silently weaken the gate. Defense in depth — protects the invariants this PR is not the owner of. |

Test-file niceties:

- `import { describe, it, expect } from "bun:test";` per CLAUDE.md §6.
- Paths resolved via `import.meta.dir` + a single relative walk; no `process.cwd()` dependency so the test runs the same whether `bun test` is invoked from the repo root or from `scripts/`.
- One `describe("release-images.yml — PR gate", ...)` block grouping W1–W5; one `describe("main.json — required status checks", ...)` block grouping R1–R2. Lets the failure output point at the right source file fast.
- SPDX header (`// SPDX-License-Identifier: MPL-2.0`) per CLAUDE.md §5.
- Pinned strings for the step names are stored in a single `const STEPS = [...]` at the top of the test file so a workflow rename only forces one edit + a clear test-failure pointing at it.

### Tier 2 — Integration (real Postgres, in the `check` CI job)

**No new file. Coverage:** none — deliberately empty.

This PR does not add a Query API op, does not change Validator behaviour, does
not change Adapter behaviour, does not migrate the schema, does not touch the
plugin sandbox runtime, and does not change any backend code path that runs
against Postgres. There is no integration boundary to assert against.

The closest analogue is the "does the source-of-truth file shape match what
the GitHub API expects?" question. That is covered out-of-band:

- `bun run rulesets:apply --dry-run` (already part of step-09's manual
  verification recipe — see below) parses the JSON and walks it through
  `scripts/apply-rulesets.ts`'s `stripMeta` + `normaliseForCompare` paths
  without hitting the GitHub API. If the JSON shape is malformed (bad
  field name, wrong array nesting) the script throws before any network
  call. This catches schema-level mistakes pre-PR-open.
- `bun run rulesets:check` (post-merge, maintainer-run) compares the
  committed JSON to the live ruleset state on github.com and exits 1 on
  drift. This is the closest thing to an integration check — but it only
  fires when a maintainer runs it, not in PR CI (running it in PR CI would
  require a token with the `repo` scope, which fork PRs don't have).

CLAUDE.md §6 declares "Every phase declares (b) integration tests covering
its Query API ops / Validator rules / Adapter behaviour against a real
Postgres." This PR has zero of those touched. The honest answer is to
declare the tier empty with rationale — not to invent a test that exercises
unrelated code paths to satisfy the form.

What replaces traditional E2E coverage for this PR:

### Manual production-bundle smoke (step 09)

Encoded so step 09 (verify-locally) has an exact recipe; not part of CI,
but part of the workflow's per-step verification per CLAUDE.md §6:

```sh
# 1. Lint the workflow YAML — catches structural mistakes (bad indentation,
#    unclosed mapping) that the bun test can't.
bun run lint

# 2. Typecheck the new test file alongside the rest of the workspace.
bun run typecheck

# 3. Run the new test file directly to confirm it passes before pushing.
bun test scripts/release-images-pr-gate.test.ts

# 4. Dry-run the ruleset apply to confirm the JSON's shape is valid against
#    GitHub's expected schema (script throws on malformed structure before
#    any HTTP call).
bun run rulesets:apply --dry-run
# Expect: a single `[dry-run] would update main-protection from main.json` line.

# 5. (Optional, requires `repo` scope) check live drift between the file and
#    the github.com state of the ruleset.
bun run rulesets:check
# Expect: `ok:    main-protection matches main.json` if the ruleset has not
# yet been re-applied to live (drift expected; maintainer will apply post-
# merge); `drift: ...` if the live state differs from the file — which is
# the intended diff this PR introduces, so a drift line is the expected
# pre-apply state.
```
