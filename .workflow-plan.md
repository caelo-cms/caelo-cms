# Implementation plan — Issue #47: Real-AI end-to-end test for the Live Edit flow

Branch: `feat/47-real-ai-livedit-e2e`
Linked roadmap item: <https://github.com/caelo-cms/caelo-cms/issues/47>

## 1. Goal

Caelo's existing Playwright suite (`apps/admin/e2e/*.browser.ts`) drives the AI
chat with a fixture provider (`registerTestProvider` + the
`x-caelo-test-provider` header path in
`apps/admin/src/routes/(authed)/content/chat/[sessionId]/stream/+server.ts`).
That catches wiring + dispatch + renderer regressions cheaply, but every
runtime failure that hit users in v0.10.16–v0.10.21 (empty AI response,
orphan locks after Stage, `set_nav_menu` JSON-Schema mismatch, missing
nav-menu primer) was only catchable with the *real* model planning the turn.
This plan ships the v0.13.0 slice the issue describes: a separate
`apps/admin/e2e-livedit/` suite that runs the chat through the real Anthropic
API (Sonnet 4.6 pinned, `temperature: 0`) and verifies the operator-visible
outcome deterministically (DOM + DB + admin-stderr) with one closing
LLM-vision verdict on the published page (per the operator's clarification in
the issue's second comment — vision is end-of-scenario only, not per-step).
A new `.github/workflows/e2e-livedit.yml` runs it on every PR + push to main
with no cost gate; the local pass on the operator's machine using the
already-provisioned API key is the hand-off check (operator's third
comment) before the workflow ships.

## 2. Scope (v0.13.0 — one PR)

- New test suite directory `apps/admin/e2e-livedit/` with its own Playwright
  config (`playwright.livedit.config.ts` at the admin app root, mirroring the
  existing `playwright.config.ts`). Per-test timeout 5 min, retries=2,
  trace/screenshot/admin-log capture on failure.
- One real-AI scenario shipped end-to-end: **Scenario 1 — Create a homepage
  from scratch + re-edit the hero headline** (single browser file
  `apps/admin/e2e-livedit/scenario-homepage.browser.ts`). Scenarios 2 + 3 in
  the issue are deferred to v0.13.1 follow-up PRs.
- Helpers module `apps/admin/e2e-livedit/helpers.ts` with:
  `waitForChatTurnComplete()` (SSE / DOM-marker based),
  `awaitStageComplete()` / `awaitPublishComplete()` (read the form-action
  response that already returns `staged.previewUrl`), `assertNoOrphanLocks()`
  (queries `chat_entity_locks` for the test chat),
  `assertNoChatRunnerDiagWarnings()` (greps the admin stderr capture for
  `[chat-runner] empty-response` and `[chat-runner] passive-response-diag`
  — both verified to exist at `packages/admin-core/src/ai/chat-runner.ts:1348`
  and `:1378`), and `verifyPublishedPageWithVision()` (the closing vision
  verdict — Playwright takes a full-page screenshot of the production URL,
  Node-side `fetch` posts it to `api.anthropic.com/v1/messages` with a
  pass/fail rubric prompt, JSON verdict gates the test).
- Global setup `apps/admin/e2e-livedit/global-setup.ts` — reuses the
  dev-owner seed from the existing `apps/admin/e2e/global-setup.ts` (extract
  the SETUP_SCRIPT into `apps/admin/e2e/_seed.ts` and import from both
  configs). The existing dummy-encryption triplet in `ai_providers` is left
  as-is: the resolver's env-var fallback at
  `packages/admin-core/src/ai/provider-resolver.ts` reads
  `process.env.ANTHROPIC_API_KEY` and bypasses DB decryption entirely when
  the env var is set, so no secret-box round-trip in tests.
- Admin process lifecycle handled by `e2e-livedit/global-setup.ts` (NOT by
  Playwright's `webServer` block — webServer's child-process stdout/stderr
  is not reachable from spec code, and the suite needs to grep it for
  chat-runner diag lines). Global setup spawns
  `bun run build/index.js` with stdio piped to per-run files under
  `apps/admin/test-results/livedit/admin.log`, exports the PID via a
  per-test-results scratch file, and the matching `globalTeardown` SIGTERMs
  it. Env carried on spawn: `ANTHROPIC_API_KEY` (set from
  `ANTHROPIC_API_KEY_E2E`), `CAELO_CHAT_MODEL_OVERRIDE=claude-sonnet-4-6-<dated-id>`,
  `CAELO_CHAT_TEMPERATURE=0`, `ORIGIN=http://localhost:4173`,
  `ADMIN_DATABASE_URL`/`PUBLIC_DATABASE_URL` from the test process env.
  The build step itself happens upstream in the npm script (one
  `bun run --filter @caelo-cms/shared build && bun run --filter @caelo-cms/admin build`
  before `playwright test --config playwright.livedit.config.ts`).
- Three changes in `packages/admin-core/src/ai` to make Sonnet-pin +
  `temperature: 0` actually take effect:
  - `provider-resolver.ts`: read `CAELO_CHAT_MODEL_OVERRIDE` (override the
    resolved model id when set + `NODE_ENV !== "production"`); read
    `CAELO_CHAT_TEMPERATURE` and store on the `ResolvedProvider` shape under
    the same guard.
  - `chat-runner.ts` (around line 1163, the `provider.generate({...})`
    call): add `temperature?: number` to `ChatRunnerOptions`; thread
    `temperature: options.temperature ?? resolved.temperature` into the
    `provider.generate(...)` call. Existing production call sites pass
    nothing → behaviour unchanged. **This is required**: the resolver's
    `temperature` setting otherwise never reaches the Anthropic API request
    body (verified by reading chat-runner.ts:1163–1170 — temperature is not
    threaded today).
  - The chat stream endpoint passes `options.temperature` through from the
    resolver if not explicitly overridden by the caller (the existing
    request-builder shape allows this in one extra line).
- Tiny test-stability change in `apps/admin/src/lib/components/chat/ChatPanel.svelte`:
  - `data-testid="chat-composer"` on the `<Textarea bind:value={composer}>`
    at ChatPanel.svelte:1569 (the existing placeholder `"Tell the AI what
    to change…"` is copy that may change).
  - `data-testid="chat-send"` on the send button.
  - `data-testid="chat-turn-status"` + `data-turn-state="streaming"` or
    `data-turn-state="idle"` on the small streaming indicator. Two states
    are sufficient: the test waits for `idle` after sending. Resist the
    temptation to enumerate `streaming|tool-running|done` — the test
    doesn't distinguish those and a wider state machine widens the
    contract surface for no payoff.
- New script `e2e-livedit` in `apps/admin/package.json` + a thin root-level
  alias in `package.json`: `bun run --filter @caelo-cms/admin e2e-livedit`.
  The 10× determinism check from the verification gate is a recipe in
  `docs/internal/e2e-livedit.md` (a one-liner that runs the suite ten times
  and counts pass/fail), not a shipped npm script — it's a manual operator
  task, not a routine CI gate.
- New CI workflow `.github/workflows/e2e-livedit.yml`:
  - Triggers: `pull_request` (all branches), `push: branches: [main]`,
    `workflow_dispatch`.
  - Concurrency: `e2e-livedit-${{ github.ref }}` with `cancel-in-progress: true`.
  - Compose isolation: spawn with
    `COMPOSE_PROJECT_NAME=caelo-e2e-livedit-${{ github.run_id }}` so the
    suite cannot collide with the existing `e2e` job's Postgres container
    if both run on the same PR.
  - Job: spin up the existing `docker-compose.yml` Postgres + Caddy
    services (admin runs as a `bun run build/index.js` host process spawned
    by the new global-setup, NOT by Playwright's `webServer` block — see §2
    above), `bun install`, build `@caelo-cms/shared` + `@caelo-cms/admin`
    first (the `#36`/`#51` lesson — explicit ordering required because we
    spawn the built output directly), run
    `bun run --filter @caelo-cms/admin e2e-livedit`, upload artifacts on
    failure: Playwright report dir, captured admin stdout/stderr,
    `chat_messages` + `chat_entity_locks` + `pages` + `page_modules` DB
    dump.
  - Secret `ANTHROPIC_API_KEY_E2E` (project-scoped, not the maintainer's
    personal key) injected as a job-level env var.
  - No cost guard. No schedule. No notification step beyond the standard
    PR check.
- New `.env.example` lines for `ANTHROPIC_API_KEY_E2E` (placeholder + the
  comment that it's used by `e2e-livedit` only).
- `.gitignore` already covers `.env`; extend it explicitly with `.env.test`
  so the local convention the issue calls for is enshrined.
- New docs `docs/internal/e2e-livedit.md`: how to run locally
  (`.env.test` setup, `bun run e2e-livedit`), how to add a scenario,
  what each assertion class covers, where the artifacts live on failure,
  and the determinism-loop recipe.

## 3. Out of scope (deferred)

- **Scenarios 2 + 3** (nav-menu + hosting page; Stage/Publish round-trip on
  footer copy) — the issue's `v0.13.1` follow-up. Helper surface is
  pre-shaped (`awaitStageComplete`, `verifyPublishedPageWithVision`) so
  the follow-up PRs are mostly the scenario files. **A v0.13.1 tracker
  issue is opened on the Caelo Roadmap board at PR-open time** (step 15)
  so the deferred scope is visible on the board, not lost in this PR's
  description.
- **Full containerised admin in a `docker-compose.test.yml` override.** The
  issue assumes admin lives in compose; today the existing `docker-compose.yml`
  has only `postgres` + `caddy-staging` + `caddy-production`, and the
  existing `e2e` CI job runs the admin as a host process. Moving the admin
  into compose for tests would be a separate refactor — not required for
  catching the regressions this issue targets. Plan keeps the host-process
  shape; the workflow is consistent with the existing `e2e` job.
- **LLM-as-judge mid-flow** (per-page semantic verdicts). Explicitly
  rejected in the body; the closing single-page vision verdict is the
  exception per the operator's second comment.
- **AI driving Playwright.** Never. Playwright is the deterministic driver.
- **Cross-provider matrix** (OpenAI/Gemini/Bedrock). PAL supports it; out of
  scope here.
- **Visual / pixel-diff regression.** Doesn't apply when content varies.
- **Cost tracking / budgets.** Not gated.
- **Scenarios that depend on shipping #45 (themes) or #46 (modules/content
  split).** Added later as those features land.
- **Production smoke tests against caelo-cms.com itself.** Manual dogfood
  covers it.

## 4. Architecture impact

### Permission layers (CMS_REQUIREMENTS §3.1, CLAUDE.md §2)
This work touches **no permission-layer behaviour**. It is test scaffolding
plus three small affordances in shipped code:
1. Two env-var-gated reads in `provider-resolver.ts`
   (`CAELO_CHAT_MODEL_OVERRIDE`, `CAELO_CHAT_TEMPERATURE`) that activate only
   when `NODE_ENV !== "production"` — never alters the production
   resolution path.
2. Three `data-testid` attributes + a `data-turn-state` attribute on
   `ChatPanel.svelte` — passive DOM hooks.
3. The existing seeded `ai_providers` row (dummy encryption triplet) is
   left untouched. The resolver's `process.env.ANTHROPIC_API_KEY`
   fallback path bypasses DB decryption when the env var is set on the
   admin process — that's the path the test relies on. No schema
   changes, no migration, no secret-box round-trip in the test suite.

### Packages touched
- `apps/admin/` — new `e2e-livedit/` directory, new Playwright config, new
  npm script, ChatPanel test-hook additions.
- `packages/admin-core/src/ai/provider-resolver.ts` — two env-var hooks
  (~6 LOC total).
- Root `package.json` — alias script.
- Root `docker-compose.yml` — **untouched.** The existing services stand up
  the suite as-is.
- `.github/workflows/` — new workflow file.
- `.gitignore`, `.env.example` — small additions.
- `docs/internal/` — new doc file.

### CLAUDE.md invariants honoured
- §6 (testing): adds a real-Postgres + real-stack test layer; no mocking the
  chat stream; coverage gates unaffected (e2e-livedit is its own job, not
  counted toward unit/integration thresholds).
- §2 ("No raw SQL anywhere"): helpers use the same `runBunInline` /
  `bun:SQL` pattern as `apps/admin/e2e/helpers.ts` — read-only queries
  scoped to test cleanup, no new Query API ops bypassed.
- §4 ("No half-finished implementations"): ships one scenario end-to-end +
  the helper surface + the workflow + docs. Scenarios 2 + 3 are deferred
  with explicit tracker ownership; no skeleton `xit("scenario 2", …)`
  placeholders.
- §5 (comments): test-only files get TSDoc on the helper exports; the
  ChatPanel `data-testid` additions need no comments.
- §8 (AI-generated code standards): preview diffs are minimal; no
  formatting churn.
- §11.A (gated ops): N/A — this work introduces no new ops.

### CMS_REQUIREMENTS cross-refs
- §17A (Skills): no skill changes.
- §17.4 (admin-only / locale gating): N/A — tests run as the dev-owner.
- §3.2 / §5 (module / snapshot architecture): Scenario 1 exercises the AI
  emitting `add_page` / `add_module` / `set_structured_set` tool calls
  through the real provider; the existing tool handlers emit snapshots
  via the normal Query API path. Scenario 1's assertions are on the
  resulting `pages` + `page_modules` + `content_instances` rows (the
  *effects* of the snapshot writes), not on the snapshot table itself —
  snapshot emission is exercised transitively rather than asserted
  directly, since that surface is already covered by snapshot/history
  unit + integration tests.
- §12 (two-DB split): test reads only `cms_admin` tables.

## 6. Risks & open questions

### Risks

1. **AI non-determinism breaks Scenario 1 intermittently.**
   - Mitigation: `temperature: 0`, pinned dated Sonnet 4.6 id, retries=2,
     structural-only mid-flow assertions (no exact-string matches on AI
     copy), substring-only DB content checks ("Caelo" + "MPL 2.0" in the
     footer module's `content_instance`). The 10× determinism loop in the
     verification gate catches over-tight assertions before merge.

2. **`data-turn-state` machine needs a client-side state already exposed.**
   - ChatPanel already tracks the streaming lifecycle internally (the
     "Reasoning ({chars})" + "🐞 Debug" affordances reference it). Open
     question: is the existing `$state` already wired such that a single
     `data-turn-state="streaming|idle"` attribute can be rendered from it,
     or does the SSE consumer need a small additional flag? Step 06
     resolves by reading ChatPanel.svelte around lines 1200–1500.

3. **Closing vision verdict adds Anthropic-API dependency in the test
   Node process.**
   - Direct `fetch` to `api.anthropic.com/v1/messages` keeps the test
     isolated from the production provider abstraction. Cost: one Claude
     vision call per scenario; trivial. Skip-gate: if `ANTHROPIC_API_KEY_E2E`
     unset, the suite fails fast with a clear message — never a silent
     skip and never a "vision passed" default (CLAUDE.md §2 "fail loudly
     with a structured error pointing at what's missing"). The helper
     retries once with 2s backoff on transient Anthropic 5xx so a brief
     edge blip doesn't waste the chat work the scenario already did.

4. **CI minutes / cost.** Each scenario is ~3–5 min including compose
   spin-up. v0.13.0 is one scenario; v0.13.1 adds two more (~15 min total
   per PR). Acceptable per operator direction ("real cost is the cost of
   correctness").

5. **`data-testid` additions to a production component.** Three attributes
   are passive (no behavioural effect). They're a thin contract surface
   but they're now part of "do not rename without updating e2e-livedit" —
   noted in `docs/internal/e2e-livedit.md`.

6. **CI Postgres collision with the existing `e2e` job.** Both workflows
   spin up `docker-compose.yml`'s `postgres` service; on a single PR
   they'd race for port 5432 and the named container `caelo-postgres`.
   Mitigation: `COMPOSE_PROJECT_NAME=caelo-e2e-livedit-${{ github.run_id }}`
   on the new workflow's compose invocations (already specified in §2).
   This prefixes the container name + isolates the network; the
   `${POSTGRES_PORT:-5432}` host port still needs to be overridden — the
   workflow sets `POSTGRES_PORT=5433` (or sources from a free port) and
   passes that into `ADMIN_DATABASE_URL` so the suite connects to the
   isolated instance.

### Open questions (resolve in step 06)

1. **Exact dated Sonnet 4.6 model id.** Use the latest dated id
   (`claude-sonnet-4-6-<YYYYMMDD>`) — confirm with `context7` /
   Anthropic SDK at implementation time. Pin in `playwright.livedit.config.ts`
   as a constant.

2. **Does the existing dev-owner seed create a homepage row, or does
   Scenario 1 start from a truly empty `pages` table?** The issue's
   Scenario 1 says "create a homepage *from scratch*". Step 06 should
   confirm by reading `apps/admin/e2e/_seed.ts` (after extraction) and
   either truncate `pages` per spec or assert pre-existing empty state.

3. **Where is the staging URL surfaced to the client?** Survey says the
   `stage:` form action returns `staged.previewUrl` (per
   `apps/admin/src/routes/(authed)/content/pages/+page.server.ts:135`).
   The helper reads it via `page.waitForResponse(/\/content\/pages/)`
   after the Stage click — verify the actual selector + response shape.

4. **Compose teardown between scenarios.** Single-scenario v0.13.0 means
   one teardown is enough; for v0.13.1's 3 scenarios, ensure
   `chat_entity_locks`, `chat_messages`, `pages`, `page_modules`, and
   `structured_sets` are truncated between scenarios so the next prompt
   starts from a known state. Helper: `resetLiveditFixtures()` running
   the same Bun + `caelo.actor_kind = 'system'` pattern as
   `resetOverlayLayoutFor`.

5. **Local-first hand-off (operator comment 3).** Step 06 must end with
   a documented `bun run e2e-livedit` pass on the operator's machine
   before the CI workflow is committed. Strategy: the workflow file lands
   in the *same* PR (the issue requires it) but the PR description
   reports the local run's success first; step 13's local-verify gate is
   the enforcer.

6. **Exact resolver → chat-runner temperature wiring shape.** The
   existing `Provider` interface returns a `generate()` method, not a
   config object — so "store temperature on `ResolvedProvider`" needs to
   pick one of two shapes:
   - **(a) Resolver returns `{ provider, model, temperature }` and the
     chat-runner spreads them into `provider.generate({...input, model,
     temperature})`.** Smallest change to the `Provider` contract; the
     resolver type grows two fields.
   - **(b) Resolver wraps the provider in a closure that injects model
     + temperature into every `generate()` call.** Smaller chat-runner
     change; harder to introspect (the runner can't see what model it's
     about to call).
   Step 06 picks (a) — it preserves the existing inspectability of the
   model id in chat-runner logs + telemetry. The plan's §5 modify-table
   entries for `provider-resolver.ts` + `chat-runner.ts` already match
   shape (a); this open-question entry just locks the decision in
   writing for the implementation step.

## 8. Test strategy

Three tiers per `CLAUDE.md §6`. The PR adds **no new Query API ops** and
**no new schema**, so Tier 2 is intentionally narrow; the bulk of the
verification work lives in Tier 1 (the load-bearing temperature plumbing
+ env hooks + vision-verdict parser) and Tier 3 (the Playwright scenario
itself, which is the whole reason for the PR).

### Tier 1 — Unit tests (`bun test`)

Three new test files, all picked up by Bun's native runner. No mocked
Postgres, no mocked Playwright — the unit tier exercises pure logic only.

#### `packages/admin-core/src/ai/__tests__/provider-resolver-test-env-hooks.test.ts` (new)

Covers AC #16 (NODE_ENV guard) + the env-parsing behaviour the
implementation introduces.

- `CAELO_CHAT_MODEL_OVERRIDE` is honoured when `NODE_ENV !== "production"`
  → the resolver's returned `{model}` field === the env var.
- `CAELO_CHAT_MODEL_OVERRIDE` is **ignored** when `NODE_ENV="production"`
  → the resolver returns the existing default (`claude-opus-4-7`).
  This is the load-bearing assertion that the test override can never
  contaminate production.
- `CAELO_CHAT_TEMPERATURE` is honoured under the same guard, parsed to a
  finite float, stored on the returned triple as `{temperature: 0}`.
- `CAELO_CHAT_TEMPERATURE="abc"` (non-numeric) → resolver throws
  `InvalidEnvConfig` with the env var name in the message
  (CLAUDE.md §2 "fail loudly with a structured error pointing at what's
  missing" — never silently fall back to default temperature).
- `CAELO_CHAT_TEMPERATURE=""` (empty string) → treated as unset; resolver
  returns no temperature.
- Both env vars only apply to the Anthropic provider — calling the
  resolver for any other provider name returns the unmodified default.
- A spy on the existing `getActiveProvider()` confirms the env-hook
  read paths are NEVER touched (early-returned) when
  `NODE_ENV="production"`.

#### `packages/admin-core/src/ai/__tests__/chat-runner-temperature.test.ts` (new)

Covers AC #17 — the load-bearing change. Uses a **spy provider** (not
fixture-replay; that's overkill for "did this one field propagate").

- Spy `Provider` whose `generate()` records its input. Given a
  `ChatRunnerOptions` with no explicit `temperature` AND a resolver
  return triple of `{provider, model, temperature: 0}`: the recorded
  `generate()` invocation has `temperature: 0` in its input.
- Given `ChatRunnerOptions.temperature: 0.7` AND resolver-supplied
  `temperature: 0`: options wins → recorded `temperature: 0.7`. (Caller
  override beats resolver default — the precedence ladder the
  implementation establishes.)
- Given neither set on options nor on the resolver: recorded input has
  `temperature: undefined` and the existing production behaviour is
  preserved.
- Negative case: a regression that drops the `temperature` field from
  the spread at chat-runner.ts:1163 causes the spy to record
  `temperature: undefined` even when the resolver supplies `0` — test
  fails with a clear message naming the chat-runner call site.

#### `apps/admin/e2e-livedit/lib/__tests__/vision-verdict.test.ts` (new)

Covers AC #18 (vision verdict fails loudly). To keep this tier
Playwright-free, the verdict-parsing logic is extracted into
`apps/admin/e2e-livedit/lib/vision-verdict.ts` and imported by
`helpers.ts`; the screenshot-driving wrapper stays in `helpers.ts`.

- Given a well-formed Anthropic response body containing
  `{"ok": true, "reason": "page renders as expected"}`: parser returns
  `{ok: true, reason: "page renders as expected"}`.
- Given `{"ok": false, "reason": "footer missing"}`: returns
  `{ok: false, reason: "footer missing"}`. (The scenario uses
  `expect(verdict.ok).toBe(true)`; a `false` verdict surfaces the
  reason in the failure message.)
- Given an Anthropic 503 response with text body, parser retries once
  with 2s backoff (verified with a stubbed `fetch` + a fake timer);
  after the second 5xx, parser throws `VisionVerdictRequestError` whose
  message includes the upstream status (e.g. `503`) AND the first 500
  chars of the body.
- Given a 2xx response with malformed JSON: parser throws
  `VisionVerdictParseError` quoting the unparseable text. Never
  defaults to `{ok: true}`.
- Given a 2xx response with valid JSON but missing the `ok` field:
  parser throws `VisionVerdictSchemaError` listing the missing field.
  Same "no silent default" guarantee.
- Given a 2xx response with `{"ok": "yes", "reason": "..."}` (non-bool
  `ok`): same `VisionVerdictSchemaError`.

### Tier 2 — Integration tests (`bun test`, real Postgres in compose)

The PR adds no new Query API ops. The one new integration boundary
worth a real-DB test is the resolver's env-var fallback against a row
that has the dummy encryption triplet — the path the e2e-livedit suite
itself relies on.

#### `packages/admin-core/src/ai/__tests__/provider-resolver-env-fallback.integration.test.ts` (new)

Real `cms_admin` Postgres from the existing test fixture; one
`ai_providers` row inserted with the same dummy-encryption shape the
e2e seed uses.

- `process.env.ANTHROPIC_API_KEY = "test-key-abc"`,
  `NODE_ENV ≠ "production"`, the dummy-encryption row exists →
  `getActiveProvider({name:"anthropic"})` returns a `Provider` whose
  resolved API key === `"test-key-abc"`. The resolver MUST NOT call
  `decryptSecret()` on the dummy bytes (assert by spy on
  `secret-box.decryptSecret`).
- `process.env.ANTHROPIC_API_KEY` unset, the dummy-encryption row
  exists → resolver throws `ProviderKeyUnavailable` (or whatever the
  existing error type is in `provider-resolver.ts:280–316`) with a
  message naming the env-var fallback as an option. Never returns a
  half-configured provider.
- A row with a *real* encrypted triplet AND `ANTHROPIC_API_KEY` set in
  env → DB key wins (the env-var fallback is a fallback, not an
  override of a configured key). This guards against the e2e seed
  accidentally overriding a real production key on a misconfigured
  install.

**Why not more integration tests?** The chat-runner's
temperature-propagation behaviour is covered at the unit tier with a
spy provider — adding a real-Anthropic integration test would burn
real API calls on every CI run for the same assertion. The full
end-to-end is what Tier 3 does once, against a real model. The
Tier 1 spy is the right shape for "did the field propagate through the
call chain."

### Coverage gates (CLAUDE.md §6)

- **Unit ≥ 90%:** the three new unit files cover every new pure-logic
  path (env-hook parsing in resolver, temperature spread in chat-runner,
  vision-verdict parser branches). The existing
  `provider-resolver.test.ts` already covers the unchanged paths.
- **Integration ≥ 80% of declared Query API ops:** unchanged — this PR
  adds zero new ops. The one new integration test covers the resolver's
  env-var-fallback boundary that the e2e suite depends on.
- **E2E:** every numbered AC has either an assertion in
  `scenario-homepage.browser.ts` or a structural check (file/script
  presence) the workflow itself enforces. The mapping above is exhaustive
  for the in-scope ACs (#1–#13 + #16–#18); ACs #14, #15, #19 are
  process gates owned by step 13 / step 15.

### What this test strategy catches

A reviewer reading this section should be able to predict:

- A regression that drops the `temperature` field from the chat-runner
  spread → caught by the Tier 1 spy-provider test before the spec
  runs.
- A regression that flips the env-hook guard from
  `NODE_ENV !== "production"` to `NODE_ENV === "development"` → caught
  by the Tier 1 env-hooks test (would fail under `NODE_ENV="test"` or
  `NODE_ENV="ci"`).
- A regression in the resolver that calls `decryptSecret()` against the
  dummy bytes when an env-var fallback exists → caught by the Tier 2
  integration test's spy on `decryptSecret`.
- A v0.10.17-class regression (AI returns empty turn but the runner
  silently writes an empty assistant row) → caught by the Tier 3
  `[chat-runner] empty-response` log assertion.
- A v0.10.19-class regression (orphan locks after Stage) → caught by
  the Tier 3 `chat_entity_locks` assertion.
- A v0.10.20-class regression (`set_nav_menu` JSON-Schema mismatch) →
  caught by Scenario 2 in v0.13.1 (out of scope here; the helper
  surface is ready).
- A v0.10.21-class regression (missing nav-menu primer) → caught by
  the Tier 3 `[chat-runner] passive-response-diag` log assertion AND
  Scenario 2's nav-menu DB checks (v0.13.1).
- A regression that makes the closing vision check silently default to
  pass (e.g. `catch { return { ok: true } }`) → caught by the Tier 1
  vision-verdict parser tests (negative cases throw, never default).
- An AI-non-determinism issue where Scenario 1 passes 8/10 times →
  caught by the operator's manual 10× recipe before any false-green
  PRs merge.
